# One-Time Semantic Adequacy Certification (Spec → E2E Assertions)
**Status:** Draft / Operational Plan  
**Owner:** Connor (final arbiter of spec intent)  
**Scope:** One-time certification pass to confirm that each behavior contract in `specs/contracts/` is *well-asserted* by E2E tests in `test/tests/`.  
**Non-Goal:** Making E2E tests pass against the current implementation. Tests may fail. They MUST compile.

---

## 0) Why This Exists
We want to reach a point where we can say:

> “For every normative contract in the specs, there exist E2E tests with explicit assertions that would fail if Naia violates that contract.”

This makes the later “make tests pass” phase honest: failures become implementation gaps, not missing test coverage.

---

## 1) Current Capabilities We WILL Reuse (No Reimplementation)

### 1.1 `naia-specs` Commands (existing)
We already have a Rust CLI at:  
`cargo run -p naia-specs -- <cmd>`

Key commands used by this plan:
- `registry` — contract ID registry
- `coverage` — confirms every contract has at least one annotated test
- `traceability` — emits contract ↔ test mapping matrix (generated artifact)
- `packet <id>` — emits a “contract review packet” (spec excerpt + linked tests + label index)
- `adequacy --strict` — reports mapping gaps: missing tests, missing labels, missing obligation mappings

(These commands are described as core workflow in the specs README.)  

### 1.2 How Adequacy/Labeling Works Today (observed behavior)
The spec tool recognizes labeled assertions by scanning test function bodies for either:
- `scenario.spec_expect("...")`, or
- `scenario.expect_msg("...")`

Labels are associated to a contract if they begin with:
- `<contract-id>.` (e.g. `messaging-04.t1: ...`) OR
- `<contract-id>:` (e.g. `connection-01: ...`)

Obligation mapping exists when the spec contains an **Obligations** section with IDs like `t1`, `t2`, etc.  
When obligations exist, the tool expects each obligation to be covered by at least one label using:
- `<contract-id>.<tN>` OR `<contract-id>.<tN>: ...`  
(and/or the `"<contract-id>: <tN>"` family if supported by the tool configuration)

### 1.3 Generated Artifacts (existing convention)
`specs/generated/` is already used for tool outputs like:
- `TRACEABILITY.md` (from `traceability`)
- other tool outputs (registry/bundle/etc.)

This plan will add more derived outputs under `specs/generated/` without changing existing behavior.

## 2) Definitions

### 2.1 “Mechanical Adequacy”
A contract is mechanically adequate when:
- It has at least one annotated test (`/// Contract: [id]`), AND
- It has at least one recognized label (`spec_expect` or `expect_msg`) for contract-level coverage, AND
- If obligations exist, each obligation has at least one label mapped to it.

This is what `naia-specs adequacy --strict` checks.

### 2.2 “Semantic Adequacy”
A contract is semantically adequate when:
- A human-level reviewer (LLM or Connor) can trace each MUST/MUST NOT behavior (or each obligation tN) to at least one E2E assertion whose predicate genuinely tests the behavior, with observable evidence.

**Semantic adequacy forbids “paper labels”:**
- Labels whose predicate checks unrelated conditions
- “Logical inverse” labels (e.g. labeling mismatch behavior inside a success-path predicate)
- Always-true predicates used solely to satisfy tooling

### 2.3 “Certification”
Certification means:
- Mechanical adequacy is complete (strict adequacy has no missing tests/labels/obligation mappings)
- Every contract has a semantic review verdict of PASS (as defined below)
- Artifacts are saved for the record (one-time; not CI)

## 4) The One-Time Certification Workflow (End-to-End)

### Phase 0 — Preconditions (fast sanity)

Run:

```bash
cargo run -p naia-specs -- validate
cargo run -p naia-specs -- registry
cargo run -p naia-specs -- coverage
cargo test -p naia-test --no-run
```

If `--no-run` fails, fix compilation first. (Passing tests not required.)

### Phase 1 — Mechanical Adequacy Burn-Down (tool-driven)

Goal: `naia-specs adequacy --strict` has:

* Missing tests: 0
* Missing labels: 0
* Missing obligation mappings: 0

Loop:

1. Run:

   ```bash
   cargo run -p naia-specs -- adequacy --strict
   ```
2. For each item, use:

   ```bash
   cargo run -p naia-specs -- packet <id>
   ```
3. Fix in smallest possible way:

   * Prefer renaming an existing meaningful `expect(...)` to `spec_expect("<cid>: ...")`
   * If obligations exist, add `spec_expect("<cid>.<tN>: ...")` on real predicates
   * Do NOT add “paper predicates”

Stop when strict adequacy is clean.

> Note: This phase is purely about ensuring the tooling can trace “something asserted.” It does not guarantee semantic adequacy.

### Phase 2 — Semantic Adequacy Fan-Out (LLM-driven)

This is the core “why we built the tooling” pass.

#### 2.1 Generate Worklist + Packets

1. Generate traceability (optional but useful for navigation):

   ```bash
   cargo run -p naia-specs -- traceability
   ```
2. Create the full worklist by reading the registry + packet metadata:

   * For each contract in the registry:

     * record spec file + tests from packet output
     * record whether obligations exist
3. Generate packet files:

   ```bash
   mkdir -p specs/generated/packets
   cargo run -p naia-specs -- packet <id> > specs/generated/packets/<id>.md
   ```

#### 2.2 Fan-Out Review Protocol

Split contracts into chunks (recommended: 15–25 per reviewer agent). Each reviewer MUST:

For each contract packet:

1. Read **Spec excerpt** and list all behaviors:

   * If an Obligations section exists: treat each `tN` as mandatory
   * Otherwise: identify distinct MUST/MUST NOT behaviors and decide if obligations should be recommended
2. Inspect **Tests excerpt** and **Assertion Labels**:

   * Verify at least one labeled predicate truly asserts each obligation/behavior
3. Produce verdict:

   * PASS: evidence exists for each obligation/behavior
   * FAIL: missing, weak, or paper assertions; recommend minimal edits

Output per contract MUST be written as:
`specs/generated/reviews/<id>.json` using the schema in §3.3.

#### 2.3 Evidence Standard (what counts as PASS)

A PASS MUST include:

* At least one asserted_behavior entry
* Each asserted_behavior MUST cite:

  * label string, and
  * file:line range, and
  * what observable signal is checked

A PASS MUST NOT rely on:

* “This is implied”
* “Logical inverse”
* “Would happen in real life”
* comments without predicates

### Phase 3 — Apply Review Fix Queue (manual, minimal diffs)

Use the summary dashboard to fix FAILs:

* Prefer: strengthen existing tests with better assertions
* Sometimes: add obligations to the spec (only if it clarifies multiple distinct behaviors)
* Sometimes: add a new small E2E test when a behavior is untested

Re-run Phase 2 packet generation for changed contracts and re-review only those.

## 5) Operational Guidance (Efficiency Rules)

### 5.1 Batch by Test File to Minimize Diff Thrash

When many contracts map to the same file (e.g. `13_client_events_api.rs`), do:

* one focused pass that adds real labels to existing meaningful predicates
* avoid sprinkling single labels across dozens of files in tiny commits

### 5.2 Don’t Over-Obligate Specs

Obligations are a tool. Add them when:

* a contract bundles multiple independent behaviors
* reviewers consistently misinterpret what’s being asserted
* it’s hard to map to a single crisp assertion

### 5.3 “Weak PASS” is not allowed

This process exists to certify, not to soothe.
If it’s not genuinely asserted, it’s FAIL.

## 6) Known Unknowns / Potential Gaps (Call These Out Early)

These items must be confirmed when implementing this plan:

1. **Packet output options**: Does `naia-specs packet` support writing directly to a file via flag?
   If not, shell redirection is sufficient.
2. **Line numbers in packets**: If packets don’t include line numbers, reviewers must open the referenced test files to cite file:line evidence.
3. **Spec “Test Obligations” section**: The README describes a spec section listing test names. It is unclear whether the current tooling enforces/uses this section or if it’s aspirational.
   This plan does not depend on it; it relies on `/// Contract: [id]` + label scanning.
4. **Reviewer execution environment**: This plan assumes Claude Code can:

   * run `cargo run ...` commands
   * write JSON files deterministically
   * fan-out sub-agents
     If agent fan-out is limited, run chunks sequentially; the schema still holds.

---

## 7) Exit Criteria (One-Time Certification Complete)

Certification is complete when:

1. `cargo run -p naia-specs -- adequacy --strict` reports no missing tests/labels/obligations.
2. Every contract in the registry has a corresponding review JSON.
3. Summary shows:

   * PASS: 236
   * FAIL: 0
4. Connor spot-checks a random sample (recommended 10 contracts) for sanity.
