## Executive summary

The mechanical deliverables of the migration are complete: 8 grouped feature files exist, step bindings are organized by purpose, the legacy_tests directory is gone, the Sidequest tooling (S.1/S.2/S.3) is implemented, and 172/172 non-deferred NPA scenarios pass. But the spirit is not complete. The migration created a two-tier coverage landscape — 172 real scenarios and 152 placeholder stubs — and stopped there. The plan's success criteria on binding LOC are not met, Scenario Outline usage is zero, 130 of 217 contract IDs have no real test logic, and the committed run artifacts are stale. The codebase is not yet "pristine with excellent test coverage asserting all Naia behavior works."

### 3.3 Deferred categories — what they actually need

Not all 152 deferred stubs are equivalent. There are three distinct categories:

**Category A — Genuinely policy-only (truly untestable, ~20 stubs):**
These are meta-policy contracts about test authorship practices, log non-assertion rules, and documentation conventions (common-03 through common-14 mostly). They document framework philosophy, not observable behavior. The `@PolicyOnly` tag is appropriate. No test needed.

**Category B — Integration-tested elsewhere but not in BDD (~35 stubs):**
The integration tests in `contract_tests/integration_only/` already assert these behaviors. Examples:
- messaging-01 (user errors return Result): `messaging_01_user_errors_return_result()` in `03_messaging.rs` tests this
- messaging-16 (reliable fragmentation): `messaging_16_reliable_fragmentation_allowed()` exists
- connection-12 (auth disabled): `auth_disabled_connects_without_auth_event()` exists and passes
- entity-delegation-01/02/03/04/05: all covered by `10_entity_delegation.rs` tests

These stubs could be upgraded to real BDD scenarios immediately — the product behavior is proven, the step bindings largely exist or need only minor extensions.

**Category C — Genuinely missing coverage (~97 stubs):**
No real assertion anywhere (no integration test and no BDD scenario). Examples:
- connection-04 (reject path client receives RejectEvent) — actually tested by `server_reject_connection_produces_reject_event()`, so this is Category B
- connection-16/17/18/19 (capacity reject, heartbeat timeout): integration tests exist
- entity-authority-02 through 15: no dedicated tests anywhere
- server-events-00 through 13: minimal coverage
- observability-01 through 10: mostly untested
- entity-ownership-05 through 14: some integration tests, some not

**The actionable breakdown:** approximately 20 stubs are genuinely untestable (Category A), 35–50 have existing integration test coverage that could be surfaced as BDD (Category B), and 80–100 represent genuine missing test coverage (Category C).

### 3.4 Highest-risk missing coverage

The following contract IDs have neither BDD scenarios nor integration tests, and cover behaviors that could silently regress:

1. **`entity-authority-*` (all non-trivial ones):** The `entity_auth_status.rs` state machine had 15 `todo!()` panics at T0.1 audit time (now reduced to 1 per the codebase audit). The state machine had those paths replaced with `unreachable!()` with documented invariants — this is now correct. No authority state machine tests exist in BDD.

2. **`server-events-00` through `server-events-13`:** Server-side event API is a primary consumer interface. Only events-07 and events-09 have real BDD coverage. The AuthEvent, DisconnectEvent, ErrorEvent, and RoomEvent sequences are untested in BDD.

3. **`client-events-*`:** The client event stream (InsertComponentEvent, UpdateComponentEvent, RemoveComponentEvent, InsertEntityEvent, RemoveEntityEvent) has BDD stubs but no real BDD assertions. The integration tests in `replicated_resources.rs` cover resources but not component events on regular entities.

4. **`messaging-20` (entity property buffer cap):** The FIFO cap was implemented (`PER_ENTITY_WAITLIST_CAP=128`, `VecDeque` for FIFO), but the BDD Scenario for it is still `@Deferred`. The integration test was added to `03_messaging.rs` and passes, but there is no BDD scenario verifying the 128-message eviction boundary behavior via the step-binding interface.

5. **`observability-*`:** RTT, jitter, packet loss, and bandwidth metrics are entirely untested outside of connection-smoke tests. The `05_observability_metrics.spec.md` contract (318 lines) has no BDD coverage.

## Section 4: Step binding catalog gaps

### 4.1 Median LOC is 14, not ≤8

The plan set ≤8 as the goal. Current state:

```
Count:     264 bindings
Median:    14 LOC
p25:        8 LOC
p75:       19 LOC
p90:       24 LOC
Max:       97 LOC
≤8 LOC:   69/264 (26%)
>8 LOC:  195/264 (74%)
```

The 97-LOC outlier and the 19 LOC at p75 indicate that many bindings still contain inlined scenario logic rather than composing helpers. Phase B achieved a 3.9% LOC reduction; the goal required a ~65% reduction that was deferred to Phase D's new bindings being written lean. Those new bindings were mostly one-liner stubs.

### 4.2 Scenario Outline: zero uses

The plan promised "Aggressive use of Background: and Scenario Outline:" with "Scenario Outline currently at zero usage." This remains zero. The messaging-channel matrix is the canonical candidate:

- messaging-07/08 (ordered reliable under reordering/jitter)
- messaging-09 (unordered reliable)
- messaging-10/11 (unreliable variants)
- messaging-12 (sequenced reliable)

These are near-identical scenarios differing only by channel type. A `Scenario Outline:` with an `Examples:` table would cover all 6 with 1 scenario definition and would shrink the deferred stub count significantly while driving the median binding LOC down (parameterized steps are typically 4–6 lines).

### 4.3 The `given/state.rs` monolith

At 48,589 bytes, `given/state.rs` is 10× the size of any other file in the catalog. It contains precondition setup that arguably belongs in `given/setup.rs` or `world_helpers.rs`. This isn't a correctness issue but impedes discoverability and contradicts the "greppable, ambiguity-free catalog" north star.

## Section 6: Integration test quality issues

### 6.1 `00_common.rs` — policy stamps not assertions

The 7 tests in `00_common.rs` have names like `tests_do_not_assert_on_logs()` and `framework_invariant_violations_are_internal_bugs()`, but their bodies all follow the same pattern:

```rust
// Connect a client, then:
scenario.spec_expect("common-07.t1: Tests assert on events/state, not logs", |ctx| {
    let connected = ctx.client(client_a_key, |c| c.connection_status().is_connected());
    let user_exists = ctx.server(|s| s.user_exists(&client_a_key));
    (connected && user_exists).then_some(())
});
```

This asserts "a client connected" — not the policy stated by the contract ID. `common-07` says tests must not assert on logs; the test demonstrates proper assertion style but doesn't prove anything about log-non-assertion. These pass trivially and cover no observable product behavior. They are documentation in Rust form, not regression coverage.

This is not a blocking issue (the tests pass and add no false confidence about specific behaviors), but they should be noted as policy-documentation tests rather than behavioral assertions. The BDD stub for `common-07` in `00_foundations.feature` is `@Deferred @PolicyOnly` — correctly classified — but the integration test version isn't labeled the same way.

### 6.2 `09_entity_ownership.rs` — minimal coverage

Only 3 tests exist for entity ownership:
1. `client_owned_entity_migrates_to_server_owned_delegated` — ownership migration
2. `private_entity_owner_retains_across_scope_changes` — private entity retention
3. `owner_retains_entity_when_non_owner_leaves_scope` — non-owner despawn

The contract spec (`08_entity_ownership.spec.md`, 353 lines) defines 14 contract IDs. The 8 deferred BDD stubs for ownership-05/06/07/09/10/11/13/14 have no real coverage: no BDD scenario, no integration test. Specifically:
- **ownership-06** (client-owned entity ownership migrates on disconnect): Not tested. Owner disconnect behavior has been the source of state cleanup bugs historically.
- **ownership-10** (server cannot directly modify client-owned entity): Not tested. This is a critical authority invariant.
- **ownership-13/14** (owner change events, concurrent operations): Not tested.
